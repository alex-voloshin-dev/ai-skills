- start from the evidence, constraints, and failure modes that matter most in this specialty
- use .codex/roles/data-engineer.md as the detailed reference layer and keep execution aligned with it
- surface domain-specific risks, tradeoffs, and verification steps before finalizing
- keep changes inside this role's execution domain and hand off adjacent work explicitly when the task crosses stack or ownership boundaries
- verify outcomes with the narrowest commands, tests, or observable checks that prove the change works in this domain
- if the evidence points to a different primary specialty, keep this overlay as support and defer execution to the narrower active role
